Test your integration with `rmf_test_` keys before switching to live keys for production.

## Environment summary

| | Test | Live |
|---|------|------|
| **Key prefix** | `rmf_test_` | `rmf_live_` |
| **Base URL** | `https://api.remitflex.com/v1` | `https://api.remitflex.com/v1` |
| **Created in** | Dashboard → API Keys | Dashboard → API Keys |

<Note>
  Environment is encoded in the API key prefix. There is no separate sandbox hostname.
</Note>

## Local development

Run the API locally and point your client at it:

```bash
export API_BASE="http://localhost:4000/v1"
export REMITFLEX_API_KEY="rmf_test_YOUR_KEY"
```

Create a test key in the dashboard with scopes: `fx:read`, `transfers:read`, `transfers:write`, and optionally `collections:read` / `collections:write`.

## Test a rate lookup

```bash
curl -s "$API_BASE/rates/USDC/EURC" \
  -H "Authorization: Bearer $REMITFLEX_API_KEY" | jq .
```

## Test a swap (dry run flow)

```bash
curl -s -X POST "$API_BASE/swaps" \
  -H "Authorization: Bearer $REMITFLEX_API_KEY" \
  -H "Idempotency-Key: $(uuidgen)" \
  -H "Content-Type: application/json" \
  -d '{
    "from": "USDC",
    "to": "EURC",
    "amount": 1,
    "destinationAddress": "YOUR_SOLANA_ADDRESS"
  }' | jq .
```

Deposit USDC to the returned `depositAddress` before `expiresAt` to complete the swap on-chain.

## Test a payment route

```bash
# List supported chains
curl -s "$API_BASE/payment-routes/chains" \
  -H "Authorization: Bearer $REMITFLEX_API_KEY" | jq .

# Create a route (requires collections:write)
curl -s -X POST "$API_BASE/payment-routes" \
  -H "Authorization: Bearer $REMITFLEX_API_KEY" \
  -H "Idempotency-Key: $(uuidgen)" \
  -H "Content-Type: application/json" \
  -d '{
    "name": "Test route",
    "originChainId": 792703809,
    "destinationChainId": 792703809,
    "destinationCurrency": "0x...",
    "destinationAddress": "0x..."
  }' | jq .
```

See [Payment routes](/api-reference/overview) in the API reference for full request fields.

## Idempotency in tests

Always send `Idempotency-Key` on mutating requests when using API keys. Retrying with the same key returns the cached response — useful for simulating network failures in integration tests.

## Security

<Warning>
  Do not log or commit live keys (`rmf_live_`) to version control. Test keys should also be kept out of public repositories.
</Warning>

Use separate keys per environment and per service. Revoke unused keys from the dashboard.
