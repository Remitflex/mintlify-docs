Before you create a [payment link](/products/payment-links) or [payment route](/products/payment-routes), check what Remitflex supports. Responses use **network names** and **token symbols** — not chain IDs or contract addresses.

**Scope:** `collections:read`

For the full curated table, see [Corridor matrix](/concepts/corridor-matrix).

## One call is enough

```bash
curl https://api.remitflex.io/v1/payment-routes/chains \
  -H "Authorization: Bearer $REMITFLEX_API_KEY"
```

Returns three things:

| Field         | What it answers |
| ------------- | --------------- |
| `origins`     | Networks payers can send from, with accepted `tokens` |
| `settlements` | Where you receive — **Base** (USDC, USDT) or **Solana** (USDC, USDT, EURC) |
| `routing`     | For each origin token, which settlement networks and tokens work |

Payment links use the same shape at `/collections/chains`.

<Note>
  Prefer `/chains` over `/corridors` or `/currencies` — those return overlapping data. Always call discovery at runtime; do not hardcode this page.
</Note>

`origins[].tokens` can include symbols that are **not** in `routing`. Only paths in `routing` are curated corridors — see [Corridor matrix](/concepts/corridor-matrix).

Use `key` from `origins` / `routing` as `originChainKey`, and `key` from `settlements` as `destinationChainKey`.

## Create a route

```bash
POST /v1/payment-routes
{
  "name": "Tron · USDT → Solana · USDC",
  "originChainKey": "tron",
  "destinationChainKey": "solana",
  "destinationCurrency": "USDC",
  "destinationAddress": "YourSolanaWallet..."
}
```

On Solana you may also settle to **EURC** when that destination appears in `routing`.
